serialize / unserialize 的基本行为

serialize 不会把引用这个对象的变量名序列化进去， 因为 serialize 本来就是用来传输数据，供服务端处理后根据类定义生成新的对象， 而不是用来传输一个具体的“变量”。

举个例子，如果序列化得到的是这样一段对象字符串： 

O:4:"User":3:{s:8:"username";s:5:"guest";s:9:"signature";s:5:"hello";s:8:"is_admin";i:0;}

当这串字符串传给 unserialize 时，PHP 会按照对象名 User 去寻找对应的类定义：

  • 如果找到了类定义，unserialize 会按照这个类定义来还原这个对象， 更准确地说，是“克隆”一个对象实例： 不是 new，因此不会调用 __construct； 同时它不会去克隆方法，只克隆数据（属性），因为方法本身就属于类，克隆方法没有意义。
  • 如果没有找到类定义，默认行为是生成一个不完善的对象 __php_Incomplete_Class。这种对象“只有 __destruct 方法，别的没有”， 也就是说正常的业务方法都指望不上。

此外，如果题目设置了 AutoLoad，PHP 可能会尝试在本地文件系统里面去找这个类的定义所在文件并加载， 这就给了出题人/攻击者一个机会，在自动加载的过程中“带点私货”， 让 PHP 去加载一份恶意代码文件。

unserialize 崩溃的典型场景

总结 unserialize 崩溃并返回 false 的两个常见场景：

  • 变量类型写错：比如本来应该写 s（string），结果写成了 k， 由于 PHP 不认识 k 这种类型标识符，解析会直接失败。
  • 序列化结构错乱：比如大括号没有闭合、引号数量不对等，导致整体结构非法。

在这些情况下，unserialize 会直接返回 false，而不是给你一个半成品对象。

__wakeup 的角色

另外一个在反序列化里非常重要的角色是 __wakeup()。 通常情况下，用户传入的序列化对象并不可靠，这时候就会自动调用 __wakeup 来把可能的不合法数据修正掉，努力让反序列化后的对象变得更安全一些。

换句话说，__wakeup 经常被用来做：

  • 重新初始化或重置敏感属性；
  • 对反序列化出来的数据做类型/范围校验；
  • 在检测到异常时抛错、打日志，或者干脆中止后续逻辑。

如果一个类根本没有实现 __wakeup，那用户就相当于获得了“胡作非为”的权限： 可以直接通过构造序列化字符串，任意控制对象的内部属性状态。

CVE-2016-7124：属性数量不一致的问题

在 CVE-2016-7124 之前，如果序列化数据的属性数量和声明的数量不一致， PHP 的处理逻辑有一些“微妙”的地方：

  • 如果序列化数据声明的属性数量多于实际给出的属性数量， 反序列化会把那些没有被提供的属性设置为 null；
  • 如果序列化数据声明的属性数量少于实际给出的属性数量， 解析器在解析到“足够数量”的属性后，会直接奔向结尾的大括号，结束解析， 后面多出来的那一部分就被粗暴忽略了。

CVE-2016-7124 的一大用点，就是当声明的属性数量和实际数量不对板时， 解析器会在 __wakeup() 之前抛出错误，终止整个解析流程。

但是！在因为错误导致解析流程终止时， 受这次漏洞影响的那个实例并不会去销毁对象。 注意，这里说的不是调用 __destruct。

在实际场景中，__destruct 反而经常是攻击目标，因为里面经常写着一些敏感的收尾逻辑， 比如保存数据到数据库、写日志或者清理一些重要资源。 如果能在一个“半解析状态”的对象上错误地触发 __destruct， 很可能会出现意料之外的安全问题。

这一点从侧面也说明了：这是 PHP 在实现上的一个失误，所以它才会上升为一个 CVE。

在 7124 之后，PHP 修复了整个流程：

  • 先接收完整的序列化数据并尝试解析；
  • 如果发现序列化对象的属性数量不对板，就直接抛出异常，终止解析进程；
  • 终止解析时，会在内存中把刚才那个解析了一半的对象正确抹除掉。

强调了一个容易混淆的点： 7124 关注的是属性数量，不是属性值的长度。 属性值长度不对板并不会触发 7124，在整个 CVE 前后， 属性值长度如果偏小导致结构错乱，往往是直接崩溃， 除非截断之后剩下的部分刚好还能组成一个合法的序列化结构。

关于字符串逃逸

至于字符串逃逸： 由于字符串逃逸并不违反 PHP 的设计理念，也不是解析器实现的 bug， 所以它不是 CVE，以后大概率也不会因为它被单独报一个 CVE。

字符串逃逸整体看起来比较迷惑，但实际上原理没有很难， 它其实就是通过精心构造长度，让原来的序列化结果里的一部分被挤出去或者被吞掉， 从而让后续部分的结构变成你希望的样子。 这方面网上已经有很多详细教程，所以就不再展开，只是补充了一些限制条件。

示例： 

s:10:"{占位符}";{占位符}=1234567890";s:7:"payload";s:3:"val";

这一类构造在长度设置正确的情况下是可以正常被解析的； 但如果你把占位符换成： 

{占位符}=12345678901";s:7:"payload";s:3:"val";

那么在解析器读取完声明长度 10 个字节后，指针会停在 0 的位置， 下一字节是 1，而 1 并不是一种类型标识符。 这时候解析器就会发现序列化结构已经非法，于是报错并删除对象。

最后强调两个特别重要的结构性限制：

  • 字符串逃逸时，即便你可以“按自己邪恶的想法”吞/吐字符串， 解析器在继续读取时，整个序列化结果的结构不能被破坏到非法， 否则就会直接崩。
  • 序列化结果中所有属性必须是偶数个，呈现键值对格式。 这在构造复杂 payload 时非常关键，如果属性总数变成奇数，整个结构就不再是合法的键值对序列了。

小结

把要点串一下，可以得到这样几个结论：

  • serialize/unserialize 本身并不“邪恶”，真正危险的是把不可信输入直接喂给 unserialize；
  • __wakeup() 是反序列化阶段的防线之一，如果缺失或者写得不严谨，就很容易被利用；
  • CVE-2016-7124 体现了实现细节对安全边界的影响，特别是对象在错误路径上的生命周期管理；
  • 字符串逃逸不是 Bug，而是一种“边缘玩法”，利用时必须同时满足长度和整体结构的合法性约束；
  • 像 __php_Incomplete_Class、unserialize 直接返回 false 这些看起来很“实现细节”的东西， 在做题和挖洞时经常会变成关键线索。

理解这些细节之后，再去看各种 php 反序列化的题目或者真实漏洞的 PoC， 会更清楚每一步到底在和 PHP 的哪个机制博弈，也更容易看出出题人到底想你踩在哪个坑上。
