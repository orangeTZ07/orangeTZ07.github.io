1. 为什么在 Arch 上常见的是 MariaDB

我在 Arch Linux 里折腾数据库时，最常遇到的就是 mariadb。我自己一开始是抱着“先学 MySQL”的目标来的， 但真正上手后发现 MariaDB 完全可以让我把 SQL 与权限体系先练扎实，语法和操作习惯都很接近。

容易误解的地方： “在 Arch 上安装 mysql 会安装 mariadb”这个说法容易被理解成严格的一一映射。 更准确的说法是：Arch 生态里通常直接安装 mariadb 作为 MySQL 兼容实现， 实战里大家大多按 MariaDB 文档走。

2. 安装、初始化、启动与安全配置

我现在更习惯这样走流程（先初始化，再启动服务）：

sudo pacman -S mariadb
          sudo mariadb-install-db --user=mysql --basedir=/usr --datadir=/var/lib/mysql
          sudo systemctl enable mariadb
          sudo systemctl start mariadb
          sudo mariadb-secure-installation

我在 mariadb-secure-installation 里会重点处理几件事： 设置数据库 root 密码、移除匿名用户、禁止 root 远程登录（推荐）、删除测试库。

容易误解的地方： 直接 start mariadb 就能首次使用，这在很多环境不成立。 首次部署常常需要先执行 mariadb-install-db 初始化数据目录。
实战建议： 我不会长期用 root 做业务操作。对我来说，root 只用于管理，“应用账号最小权限”是我给自己的安全底线。

3. 用户创建与密码设置（含纠错）

我推荐直接用标准 SQL 创建账号，不要上来就硬改系统表：

CREATE USER 'orange114'@'localhost' IDENTIFIED BY 'your_strong_password';
          -- 或者后续改密码
          ALTER USER 'orange114'@'localhost' IDENTIFIED BY 'your_new_password';
容易误解的地方： 用 UPDATE mysql.user ... 直接改密码在新版本里并不推荐， 也可能因认证插件与字段变化导致不可预期问题。优先使用 CREATE USER / ALTER USER。
容易误解的地方： FLUSH PRIVILEGES 不是所有场景都必须。我使用 CREATE USER / GRANT / ALTER USER 这类语句时，权限通常会自动生效；只有我“硬改授权表”时才经常需要手动 flush。
容易误解的地方： “数据库名和表名是否大小写敏感”受系统与配置影响（如 lower_case_table_names），不是所有环境都完全一致。

我自己以前也会把数据库叫“仓库”，这个类比确实好记；但为了和代码仓库（repository）区分， 我现在写文档时会尽量统一用“数据库（database）”这个术语。

4. 数据库与权限分配

我一般会先创建数据库，再按最小权限原则授权：

CREATE DATABASE test1;
          GRANT ALL PRIVILEGES ON test1.* TO 'orange114'@'localhost';
          FLUSH PRIVILEGES;

如果是生产环境，我会把 ALL PRIVILEGES 改成明确权限集合， 例如只给 SELECT, INSERT, UPDATE, DELETE。

容易误解的地方： 用户主机部分如果写成 'orange114'@'%'，表示可从任意来源尝试连接。 这不是“不能用”，而是安全面更大。能限定 localhost 就尽量限定。
